# Telegram-бот: предварительный расчёт выплат (СВО) и заявки

Production-ready бот на **Python 3.10+**, **aiogram 3** и **SQLite**. Проводит пользователя по короткой анкете, показывает ориентировочный расчёт (в том числе оценку «цены ожидания» для ветки «документы уже поданы»), собирает контакты и согласие на обработку ПДн, сохраняет заявки в БД и при настройке дублирует их администратору в Telegram.

> Тексты и суммы в боте — **упрощённая модель для ориентира**, а не индивидуальная юридическая консультация. В интерфейсе пользователю показывается дисклеймер.

## Возможности

- Две ветки анкеты: «впервые» и «уже подали, вопрос не решается» (с вопросами о проблеме, куда подавали, сроке ожидания).
- Персистентный **FSM** в SQLite: после перезапуска бота незавершённый сценарий можно продолжить с того же шага (`wizard_state` и ответы в `leads`).
- Черновики и завершённые заявки в одной таблице **`leads`**.
- Экран результата с разбивкой по видам выплат и (для соответствующей ветки) оценкой задержки.
- Лид-форма: имя, телефон, опционально мессенджер и комментарий → согласие на ПДн → запись в БД и опционально сообщение в чат админа.

## Требования

- Установленный **Python 3.10 или новее**.
- Токен бота от [@BotFather](https://t.me/BotFather).
- Для уведомлений о заявках — числовой **ID чата** (личный или групповой), куда бот может отправлять сообщения (бот должен быть участником группы, если это группа).

## Быстрый старт

Клонировать репозиторий, перейти в корень проекта, создать виртуальное окружение и установить зависимости.

**Windows (PowerShell):**

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
copy .env.example .env
```

**Linux / macOS:**

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Отредактируйте `.env` (минимум `BOT_TOKEN`, желательно `ADMIN_CHAT_ID`). Подробнее — в следующем разделе.

## Переменные окружения

Файл **`.env`** в корне проекта (шаблон — `.env.example`).

| Переменная | Обязательно | Описание |
|------------|-------------|----------|
| `BOT_TOKEN` | да | Токен бота от BotFather. |
| `ADMIN_CHAT_ID` | нет | Числовой ID чата для копий завершённых заявок. Если не задан, заявки только в БД, в лог пишется предупреждение. |
| `CONTACT_TEXT` | нет | Текст для кнопки «Контакты» после отказа от согласия на ПДн. |
| `DATABASE_URL` | нет | По умолчанию `sqlite:///leads.db`. Поддерживается только схема **`sqlite:///`**; путь — относительный к рабочей директории при запуске или абсолютный (`sqlite:///C:/data/leads.db`). |
| `LOG_LEVEL` | нет | Уровень логирования, по умолчанию `INFO`. |

Секреты не коммитить: `.env` должен быть в `.gitignore`.

## Запуск

Запускать из **корня репозитория** (чтобы корректно подхватывались пакет `app` и путь к БД по умолчанию):

```bash
python -m app.bot
```

При отсутствии `BOT_TOKEN` процесс завершится с сообщением в stderr.

## Docker

В корне репозитория: **`Dockerfile`** и **`docker-compose.yml`**. База SQLite и FSM хранятся в именованном томе **`bot-data`** (путь в контейнере `/data/leads.db`).

1. Скопируйте `.env.example` → `.env`, укажите как минимум `BOT_TOKEN` (и при необходимости `ADMIN_CHAT_ID` и остальное). Файл `.env` не копируется в образ, подставляется при запуске.
2. Сборка и запуск:

```bash
docker compose up -d --build
```

3. Логи:

```bash
docker compose logs -f bot
```

4. Остановка:

```bash
docker compose down
```

Данные заявок сохраняются в томе; `docker compose down` **без** флага `-v` том не удаляет. Переменная `DATABASE_URL` в `docker-compose.yml` переопределяет значение из `.env` для пути к БД в контейнере.

## База данных

При старте вызывается `init_db`: создаётся файл SQLite (каталог при необходимости создаётся автоматически), включается **WAL**.

- Таблица **`fsm_storage`** — состояние и данные FSM для aiogram.
- Таблица **`leads`** — поля анкеты, контакты, флаги согласия и завершения, `wizard_state` для восстановления UI.

Индекс: `idx_leads_tg_completed` по `(telegram_user_id, completed)`.

## Проверка воронки

1. Отправить `/start`, выбрать ситуацию, пройти анкету (кнопки + ввод региона текстом).
2. На экране результата проверить корректность сумм (контрольная **базовая** сумма для дефолтных коэффициентов в коде: **14 208 154,28 ₽**; при нескольких получателях — пересчёт доли).
3. Нажать **«Получить разбор ситуации»**, заполнить имя и телефон, при необходимости пропустить или заполнить мессенджер и комментарий, подтвердить согласие на ПДн.
4. Убедиться, что в `leads` появилась строка с `completed = 1`, а при заданном `ADMIN_CHAT_ID` — пришло сообщение администратору.

## Деплой и миграции

- **Long polling** — достаточно держать процесс `python -m app.bot` под systemd, supervisord, Docker и т.п.; отдельный веб-сервер не требуется.
- Для **PostgreSQL** или другой СУБД потребуется заменить реализацию `LeadRepository` и при необходимости FSM storage, сохранив контракт методов и семантику полей модели заявки.
